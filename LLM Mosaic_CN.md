# LLM Mosaic

> 一种用大语言模型对同一主题的多份独立来源做**并列呈现、差异标注、逐条溯源**的知识组织模式。
>
> LLM Wiki 回答的问题是"关于 X，我知道什么"。LLM Mosaic 回答的问题是"关于 X，A 怎么说、B 怎么说、C 怎么说——以及他们哪里说得不一样"。
>
> 这是一份理念文件，设计意图是让你复制粘贴到你的 LLM Agent 中使用。它传达核心理念和操作模式，具体的落地实施由你的 Agent 与你协作完成。

---

## 核心理念

### LLM Wiki 做不到的事

LLM Wiki 的默认模式是"综合"——LLM 读多份资料，消化、合并、输出一份连贯的摘要。这对大多数场景是够用的。一本教材、一份年报、一套会议纪要——你只需要一个清晰准确的答案。

但有些领域，**"综合"本身就是问题**。

历史人物传记：林语堂笔下的王安石是偏执狂，李一冰笔下的王安石是理想主义改革家。你该信谁？LLM 给你一个"综合来看，王安石是一个复杂的历史人物……"——这句话没错，但它抹掉了最关键的张力。

法律判例：控方、辩方、法官对同一事实有三种叙述。LLM 综合成"本案涉及……"——你失去了视角之间的裂缝。

竞品分析：五份行业报告对同一赛道的判断可能截然相反。LLM 综合成"市场前景总体乐观，但存在不确定性"——你失去了做出差异化判断的抓手。

文学批评：五位评论家对同一部作品有五种解读。LLM 综合成"这部作品主题丰富，引发多方讨论"——你失去了思想碰撞的火花。

**共同模式：当一件事有多个独立来源的独立判断，这些判断之间的差异本身就是最有价值的信息。但传统 LLM 工作流的设计目标恰恰是消除这些差异。**

### LLM Mosaic 的解法

LLM Mosaic 的核心原则只有一条：**汇集，不编译。**

把每一份来源对同一实体的说法一字排开。保持原貌，不合并，不替读者判断。每条说法后面跟一个链接——点一下跳回原文的精确位置。

```
❌ "综合来看，乌台诗案是北宋党争的产物……"  ← LLM 替读者做了判断
✅ "李一冰说：沈括是始作俑者。……📖"
   "林语堂说：王安石的小人集团主导。……📖"
   "周文翰说：印刷文化造就的文化明星被攻击。……📖"
   "祝勇说：苦难造就艺术的前奏。……📖"
                                            ← 读者看原文自己判断
```

这不是"不信任 AI"——这是**承认有些问题的价值在于视角的并存，而非答案的统一**。

当 LLM 帮你把四个传记作者对同一事件的论述并排放在一个屏幕上，每条都链回原文——你作为人类读者获得的判断力，远超过任何一个"综合结论"能给你的。

## 架构

三层，但中间层的角色与 LLM Wiki 完全不同：

```
┌──────────────────┐      ┌──────────────────┐      ┌──────────────────┐
│   原始资料         │  →   │   Mosaic 页面     │  →   │   产出层          │
│   (原料库)         │      │   (多源并列)       │      │   (可选的输出)     │
│   只读不改          │      │   每源一段+溯源     │      │   文章/脚本/报告   │
└──────────────────┘      └──────────────────┘      └──────────────────┘
        ↑                         ↑                         ↑
        └── 每条引用可点回原文精确位置 ──┘
```

### 第一层：原始资料

你策展的多份独立来源。和 LLM Wiki 一样——LLM 只读不改。关键区别在于：你选的资料必须是**对同一主题的独立视角**，而不是递进或互补的教材章节。

什么算"独立来源"：

- 同一人物的多本传记（不同作者、不同年代、不同立场）
- 同一事件的多家媒体报道（不同媒体、不同政治立场）
- 同一案件的多方文书（控方、辩方、判决书）
- 同一市场的多份研报（不同机构、不同时间、不同假设）
- 同一作品的多个评论（不同流派、不同范式）
- 同一历史时期的多国档案（不同国家视角）
- 同一技术的多篇论文（不同团队、不同方法）

一份来源就是一个**完整的视角**——它不是拼图的一块，它是对整幅图景的独立描绘。多源放在一起，你看到的是同一幅图景的多个版本。

### 第二层：Mosaic 页面

这是 LLM Mosaic 的核心。每个实体（人物、事件、概念、判例、产品）一个页面。页面结构如下：

```markdown
## 时间线（如适用）
按时间排列的客观事实，每个事件带年谱/档案溯源

## 各来源视角
### 来源A：核心判断 + 逐段原文引用 + 📖
### 来源B：核心判断 + 逐段原文引用 + 📖
### 来源C：核心判断 + 逐段原文引用 + 📖
### 来源D：（如未涉及，诚实标注"未专门述及"）

## 关键差异
一张并排对照表，列出各来源在哪些点上说法矛盾、哪些点上互相印证
```

与 LLM Wiki 的关键区别：

| | LLM Wiki | LLMMosaic |
|---|---|---|
| **信息来源** | 通常一个领域一份主要来源 | 同一主题多份独立来源 |
| **页面结构** | 综合摘要 + 概念展开 | 各来源并列 + 差异标注 |
| **LLM 的角色** | 消化 → 综合 → 输出 | 定位 → 摘出 → 排好 → 链接 |
| **读者的角色** | 接收已提炼的知识 | 看原文自行判断 |
| **"共识"** | 追求的产出 | 不追求，差异本身就是价值 |

### 第三层：产出层（可选）

同一套 Mosaic 页面可以输出多种格式——文章、脚本、报告、日历、可视化。核心原则是：**输出时允许表达立场，但素材来源始终透明**。读者可以不同意你的解读，但每个论断都能追回原始出处。

### 辅助文件

两个特殊文件支撑 Mosaic 的运作：

**entity-index.md**：实体 × 来源矩阵。这是 Mosaic 的"总控台"——LLM 在编译任何实体前，先查这个表，知道该实体在哪些来源的哪些章节被提及。

```
| 实体 | 来源A | 来源B | 来源C | 来源D | 年谱 |
|------|-------|-------|-------|-------|------|
| 沈括 | 大量  | 多次  | --  | 少量  | 7条  |
| 王安石 | 大量 | 大量 | 少量 | 大量 | 23条 |
| 乌台诗案 | 核心事件 | 核心事件 | 少量 | 核心事件 | 15条 |
```

矩阵的价值不在于精准计数，而在于快速判断：**这个实体在哪些来源中有话可说？哪些来源完全没提？跨源矛盾集中在哪些行？**

**log.md**：只追加操作日志。记录每次 ingest（新来源入库扫描）、每次 compile（编译实体页）、每次 lint（跨页一致性检查）。

## 操作

四种核心操作：Ingest（入库扫描）、Compile（编译实体页）、Query（多源查询）、Lint（差异审计）。

### Ingest（入库扫描）

当一份新来源加入时，不是直接"读+理解+写入 wiki 页面"，而是：

1. **放入原始资料库**。格式转换（EPUB→MD / PDF→MD / 网页→MD），保持只读。
2. **全量实体扫描**。LLM 扫描新来源，提取所有被提及的实体（人物、事件、概念），记录每个实体在哪个章节出现。
3. **更新 entity-index.md**。在矩阵中新增一列（新来源），对每个已有实体标注"有/无/大量/少量"。发现新实体则加行。
4. **选择性回刷**。对 entity-index 中多源覆盖的实体，补充新来源的视角到已有 Mosaic 页面。不要求全量回刷——高价值实体优先。
5. **更新 log.md**。

关键设计：**一次扫描、矩阵记位、按需回刷**。不是每个新来源都要读完、理解完、写入所有受影响的页面。扫描阶段只做定位，编译阶段才深入阅读具体章节。

### Compile（编译实体页）

对 entity-index 中标记了多源覆盖的实体，启动编译流程：

1. **查矩阵**。定位该实体在哪些来源的哪些章节被提及。
2. **只读相关章节**。不是全本通读——按 entity-index 的定位信息只读相关段落。一本传记 10 万字，谈沈括的只有 2000 字——只读那 2000 字。
3. **按来源汇集**。每个来源摘出关键段落，标注核心判断，附 📖 链接回到原文。
4. **写差异表**。并排对比各来源在关键点上的异同。
5. **诚实标注空源**。如果某个来源完全没提这个实体，写"未专门述及"——不留白不等于不存在。一个来源选择不写某个人物，本身就是一个信息。
6. **更新 index.md 和 log.md**。

### Query（多源查询）

你对着 Mosaic 提问。LLM 不是"综合回答"，而是"分源报告"：

- "沈括在乌台诗案中扮演了什么角色？" → LLM 的回答是"李一冰认为……林语堂认为……周文翰认为……祝勇认为……。差异在于……。详见各来源段落。"
- "各传记对王安石改革的总体评价如何？" → LLM 读取各传记中王安石的论述段落，生成对比表，标注共识与分歧。

答案始终带溯源链接。你随时可以点回原文验证任何一个说法。

### Lint（差异审计）

定期让 LLM 做跨页审计。LLM Mosaic 特有的检查项：

- **差异标注完整性**：entity-index 中标记了多源覆盖的实体，是否都已经编译了 Mosaic 页面？
- **溯源链接有效性**：📖 链接是否仍然指向正确的章节？原始资料是否被移动或重命名？
- **空源诚实度**：标注为"未专门述及"的来源，是否需要重新扫描确认？（可能是扫描时的漏报）
- **矛盾标注**：两个来源在同一事实上说法矛盾，是否在差异表中被显式标注？还是各自列各自的、读者看不出来？
- **新源回刷进度**：最近入库的来源，对高价值实体的回刷进度如何？哪些关键实体还没有补上新来源的视角？

### 新增来源后的完整工作流

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 新书入库  │ →  │ 实体扫描  │ →  │ 更新矩阵  │ →  │ 选择性回刷 │
│ EPUB→MD  │    │ grep关键词 │    │ 加一列     │    │ 高价值实体  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                     ↓
                                               ┌──────────┐
                                               │ 更新索引  │
                                               │ 追加日志  │
                                               └──────────┘
```

## 核心机制

### 机制 1：实体 × 来源矩阵

entity-index.md 是 Mosaic 的总控台。它的价值不在于"完整性"，而在于**让 LLM 在编译前知道去哪找、编译时只读该读的部分、新增来源时知道该补哪些实体**。

没有这个矩阵，LLM 每次编译都要通读所有来源——6.6MB 原文全塞进上下文窗口，成本爆炸且质量下降。有矩阵后，编译沈括只需要读 4 段相关章节的原文，几百字就能完成，LLM 的注意力集中在真正需要判断的地方。

### 机制 2：汇集，不编译

这是 LLMMosaic 区别于 LLM Wiki 的灵魂。

LLM Wiki 的默认姿态是"我来帮你消化这些资料，给你一个清晰简洁的结论"。这在大多数场景下是优点——你不需要看 30 页的摘要，你只需要知道重点。

但在多源互证的场景下，"消化"就是"抹平偏差"。四本传记各说各的，LLM 消化之后给你一个"综合来看"——这四个字的代价是让你再也看不到李一冰和周文翰之间最精彩的张力和分歧。

**LLM Mosaic 要求 LLM 抑制"消化冲动"**。找到原文，摘出原文，排好顺序，链接回去。不要加"综合来看""总体而言""本质上"。读者来做判断，LLM 做的是把判断所需的原始材料放到读者面前。

### 机制 3：逐条溯源

每条说法后面跟一个可点击的链接，跳回原始资料的精确位置。

为什么要精确到章节/段落，而不是"参见《苏东坡新传》"？因为"参见某本书"等于没有引用——那是整本 300 页的搜索任务。"参见第五章第二节"是一个 5 页的搜索任务。`[[来源#^block-id]]` 是一次点击。

引用的颗粒度决定了验证的成本。颗粒度越细，验证成本越低，读者越可能真的去验证。颗粒度越粗，"溯源"就变成了一种姿态而非功能。

### 机制 4：诚实标注空源

某份来源在这个实体上无话可说——不是跳过，而是显式标注"未专门述及"。

这很重要。林语堂 400 页的《苏东坡传》几乎没有写沈括——这不是偶然遗漏，这是林语堂的叙事选择。他在写一个浪漫化的苏东坡，沈括这个"乌台诗案告密者"在他的叙事框架里装不进去。**一个来源选择不写什么，和一个来源写了什么，同样构成一个视角。**

### 机制 5：并排差异表

各来源在关键点的异同，不用自然语言段落描述，用表格并排：

| 关键点    | 李一冰         | 林语堂     | 周文翰        | 祝勇     |
| ------ | ----------- | ------- | ---------- | ------ |
| 乌台诗案主因 | 沈括告密 + 台谏制度 | 王安石小人集团 | 印刷文化下明星被围攻 | 苦难造就艺术 |
| 沈括角色   | 始作俑者        | 未专门述及   | 新党红人察访     | 旧交卧底   |

表格的价值：读者一眼看到四个版本在一个维度上的位置关系。差异不是被"描述"的——是被"看见"的。

## 什么时候用 LLM Mosaic，什么时候用 LLM Wiki

| 信号 | → 用 LLM Wiki | → 用 LLMMosaic |
|---|---|---|
| **来源性质** | 教材式：互补、递进、无根本分歧 | 传记/评论/报道式：独立判断、存在分歧 |
| **你的目标** | 掌握一个领域的知识体系 | 看清同一事物的多个版本 |
| **"正确"** | 有一个相对客观的答案 | 没有单一答案，视角本身就是价值 |
| **来源数量** | 1-2 份核心来源为主 | 3+ 份独立来源 |
| **典型场景** | 学一门课、读一本书、整理内部文档 | 历史研究、判例分析、竞品情报、文学批评、多国视角 |

两者不互斥。苏东坡项目就是 LLM Wiki 的架构 + LLM Mosaic 的多源汇聚方法——三层结构来自 LLM Wiki，汇集不编译来自 LLM Mosaic。你可以在一个项目中混用两种模式。

## 为什么这行得通

### 1. LLM 在"找和排"上远超人类，在"判断"上不替代人类

翻阅四本传记寻找同一人物的所有提及——人类需要几小时，LLM 需要几秒钟。但判断李一冰和林语堂谁更可信——这是人类读者的事。LLM Mosaic 把 LLM 用在其绝对优势（定位和编排），把最终判断保留给人类。

### 2. 差异是信息，不是噪声

传统知识管理追求一致性——wiki 里的一个页面不能说"沈括既是小人又是天才"，这会让读者困惑。但历史研究恰恰需要这种"困惑"——沈括在不同传记作者眼中有截然不同的形象，这个差异本身就是关于沈括、也关于传记作者的最重要信息。LLM Mosaic 设计的页面结构天然容纳这种张力。

### 3. 矩阵做减法

当你只有 3 份来源时，管理跨源关系是手工可维护的。当来源增长到 7 份、15 份、30 份，"谁在哪说了什么"变成组合爆炸问题。entity-index 矩阵把组合爆炸压缩为一次扫描 + 一张表 + 按需回刷。新来源加入的成本是 O(n) 的扫描，而非 O(n²) 的交叉对比。

### 4. 复利来自"加一列"

每加一份新来源，已有 Mosaic 页面的价值不是"变多了一点"，而是**多了一个维度**。当你有两份来源说同一件事，那就是两个点。三份来源就是三角定位。十份来源，你开始看到"叙事光谱"——某些实体在所有来源中高度一致，某些实体在不同来源中几乎不相容。这种复利效应只有在"汇集不编译"的模式下才能累积——因为每次综合都会把之前的多维信息坍缩成一个点。

## 说明

本文档描述的是理念和操作模式，而非具体实现。确切的目录结构、页面格式、溯源精度（章节级 vs 块级）、实体分类体系——这些都取决于你的领域、你的来源类型、你使用的 LLM 和阅读工具。

所有提到的东西都是可选和模块化的。你可能只需要 3 份来源不需要 entity-index。你可能不需要产出层只在 Mosaic 页面中浏览和思考。你可能需要块级引用因为你的资料来源是 PDF 扫描件而非有目录的电子书。你的领域有自己的实体类型和差异维度——改编模板，不要照搬。

正确的使用方式是与你的 LLM Agent 分享这份文档，告诉它你的领域、你的来源、你想看清什么。共同协作实例化一个适合你的 Mosaic。这份文档唯一的工作就是传达"多源并列、逐条溯源"这个模式。你的 LLM 能搞定剩下的。

---

*LLM Mosaic 的方法论源于苏东坡研究项目的实践——四本传记 + 三册年谱，400W字，56 个实体 × 7 份来源的跨源矩阵。但它的适用场景远超历史传记：任何"多个独立视角看同一个东西"的领域，都可以用这套模式组织知识。*
